# git-commit スキル

変更を論理的な単位に分割し、意味のある日本語コミットメッセージとともにコミットする。

## 基本方針

コミットは「なぜこの変更をしたか」が伝わる単位にする。ファイル単位ではなく、**意図の単位**で区切る。

- 1コミット = 1つの論理的な変更（機能追加、バグ修正、リファクタリングなど）
- 互いに依存しない変更は別コミットに分ける
- 依存関係があるものは依存される側を先にコミットする

## 手順

### 1. 変更全体を把握する

```bash
git status
git diff HEAD
```

差分を読み、変更を**論理グループ**に分類する：
- どのファイルがどんな目的で変わっているか
- 複数ファイルにまたがっていても同一目的の変更はまとめる
- 無関係な変更が混在していれば分ける

### 2. 分割案を提示してユーザーに確認する

変更が複数の論理グループに分かれる場合、コミット分割案を提示する。

**提示フォーマット：**
```
提案するコミット分割：

1. `<メッセージ案>`
   対象: foo.nix, bar.nix
   内容: 〇〇の設定を追加

2. `<メッセージ案>`
   対象: flake.lock
   内容: flake inputs を更新
```

変更が1つの論理グループなら分割不要。そのまま進める。

### 3. コミットメッセージを作る

このリポジトリのコミットメッセージは**日本語**で書く。

**スタイル：**
- 体言止めまたは動詞止め（「〜を追加」「〜に移行」「〜を修正」）
- 何をしたかが一目でわかる具体的な表現
- prefix（feat:, fix: など）は使わない
- 50文字以内を目安に

**良い例：**
- `home/langs に java / php / lua を追加し bash-language-server を共通 LSP に追加`
- `ghostty 設定を home-manager の programs.ghostty モジュールに移行`
- `Agent Skills の idPrefix を削除し有効スキルを明示的に指定`

**避けるべき例：**
- `update`（何を更新したか不明）
- `fix bug`（英語・内容不明）
- `feat: add something`（prefix + 英語）

### 4. ステージングとコミットを実行する

分割する場合は、**1コミットずつ**順番にステージングしてコミットする。

ファイル全体をステージする場合：
```bash
git add <file1> <file2>
```

ファイルの一部（hunk）だけステージする場合は `git add -p` を使い、必要な hunk だけ `y`、不要なものは `n` を選ぶ。ただし `-p` はインタラクティブなので、代わりに `git diff` で差分を確認してから範囲を特定し、必要であれば一時的なステージング方法を検討する。

コミット：
```bash
git commit -m "メッセージ"
```

### 5. 完了確認

```bash
git log --oneline -5
```

コミット後にログを表示して、分割が意図通りになっているか確認する。

## 注意事項

- `flake.lock` の更新は単独でコミットするか、関連する input 更新と一緒にまとめる
- 設定ファイルとその依存関係（lock ファイルなど）は同一コミットにする
- ユーザーが「まとめてコミット」と言った場合は分割不要と判断してよい
